목표는 **GUI 없는 CLI 기반 블랙박스 버그바운티 자동화 도구**이고, 핵심은 **자동 공격/자동 제보가 아니라, 허가된 scope 안에서 취약점 후보를 최대한 잘 찾고 사람이 검증할 수 있게 정리하는 엔진**임.

# 블랙박스 버그바운티 자동화 도구 개발계획서 (DONZO)

## 1. 프로젝트 개요

### 1.1 프로젝트명 후보

DONZO

### 1.2 프로젝트 목표

본 프로젝트는 웹/도메인 기반 블랙박스 버그바운티 환경에서 다음 작업을 자동화하는 CLI 기반 보안 자동화 도구를 개발하는 것을 목표로 한다.

```text
1. 허가된 bug bounty scope 입력
2. in-scope / out-of-scope 검증
3. 서브도메인 및 외부 자산 수집
4. DNS 검증 및 wildcard filtering
5. 살아있는 웹 서비스 식별
6. 포트 및 서비스 탐지
7. URL, 엔드포인트, API route 수집
8. JavaScript 파일 분석
9. source map, API endpoint, secret 후보 탐지
10. archive URL 수집
11. hidden parameter 탐지
12. Swagger/OpenAPI/GraphQL 노출 탐지
13. BOLA/IDOR 후보 생성
14. SSRF/open redirect/file disclosure/subdomain takeover 후보 생성
15. nuclei/ZAP/Dalfox/interactsh 등 제한적 스캐너 연동
16. 결과 정규화
17. 중복 제거 및 finding clustering
18. 위험도 기반 우선순위화
19. evidence 저장
20. 수동 검증 queue 생성
21. report.md, JSONL, SARIF optional 산출물 생성
```

이 프로젝트는 **자동 exploit**, **자동 대량 공격**, **자동 제보 제출**을 목표로 하지 않는다. 최종 목표는 버그바운티에서 사람이 검증할 가치가 높은 후보를 자동으로 수집하고, 증거와 재현 단서를 정리하는 것이다.

## 2. 핵심 설계 철학

### 2.1 기본 방향

```text
이 도구는 “자동 공격 도구”가 아니라 “취약점 후보 수집/정리/우선순위화 엔진”이다.

핵심 가치는 다음 5가지다.

1. Scope 강제
2. Deep recon
3. API / JS / parameter 중심 분석
4. BOLA / IDOR 후보 생성
5. evidence 기반 report 자동화
```

### 2.2 SCOUT와의 관계

참고 프로젝트인 SCOUT은 firmware blob을 대상으로 evidence-anchored exploitability chain, SARIF, CycloneDX SBOM 등을 생성하는 펌웨어 분석 플랫폼이다. 본 프로젝트는 SCOUT의 “증거 중심 결과 생성” 철학을 웹 블랙박스 버그바운티 영역에 적용한다. 단, SCOUT 자체는 펌웨어 분석용이고, 본 프로젝트는 웹/API 블랙박스 버그바운티 자동화용이다. ([GitHub][1])

```text
SCOUT:
firmware blob
 → firmware analysis
 → exploitability evidence
 → SARIF / SBOM / dossier

본 프로젝트:
bug bounty scope
 → web/API recon
 → vulnerability candidates
 → evidence
 → manual verification queue
 → report.md / JSONL / SARIF optional
```

## 6. 안전 정책

### 6.1 기본 원칙

```text
모든 스캔은 authorized-only security testing을 전제로 한다.

도구는 반드시 bug bounty program scope, rate limit, 금지 테스트 조건을 준수해야 한다.

자동 exploit, 자동 제보, brute force, credential stuffing, DoS, destructive test는 기본적으로 금지한다.
```

## 9.9 Vulnerability Candidate Engine

이 프로젝트의 핵심 모듈이다. 단순히 scanner 결과를 모으는 것이 아니라, 실제 버그바운티에서 가치가 높은 후보를 직접 생성한다.

---

### 9.9.1 BOLA / IDOR Candidate Engine

OWASP API Security Top 10 2023의 API1은 Broken Object Level Authorization이며, OWASP는 object ID가 path, query parameter, header, payload에 존재할 수 있고 이를 조작해 권한 문제가 발생할 수 있다고 설명한다. OWASP WSTG도 IDOR를 사용자 입력 기반 object 직접 접근으로 인해 권한 우회가 발생하는 취약점으로 설명한다. ([OWASP Foundation][9])

#### 입력

```text
endpoints_alive.jsonl
params.jsonl
api_docs.jsonl
js_endpoints.jsonl
```

#### 탐지 기준

```text
- URL path에 object ID 존재
- query parameter에 id류 값 존재
- user/order/invoice/document/team/org/account 키워드 존재
- JSON API 응답
- 인증 필요 endpoint로 추정
- GET/POST/PUT/PATCH/DELETE method 분류
- producer endpoint와 consumer endpoint 연결 가능성
```

#### 후보 예시

```json
{
  "candidate_type": "BOLA_IDOR",
  "endpoint": "GET /api/v1/orders/{order_id}",
  "object_key": "order_id",
  "method": "GET",
  "risk_reason": [
    "object_id_in_path",
    "user_owned_resource_keyword",
    "returns_json"
  ],
  "verification": "manual_two_account_test_required",
  "auto_exploit": false
}
```

#### 수동 검증 템플릿

```text
1. 테스트 계정 A와 테스트 계정 B를 준비한다.
2. 계정 A에서 object id를 획득한다.
3. 계정 B의 정상 세션에서 동일 object id 접근 가능 여부를 확인한다.
4. 응답에 소유자 정보, 주문 정보, 문서, 결제 정보 등 민감 데이터가 있는지 확인한다.
5. 프로그램 scope와 testing policy를 재확인한다.
6. 실제 제보 전 요청/응답 증거를 정리한다.
```

---

### 9.9.2 SSRF Candidate Engine

#### 탐지 기준


```text

```

Interactsh는 out-of-band interaction을 수집하는 server/client 도구이며, 외부 interaction을 유발하는 취약점 탐지에 사용된다. 본 프로젝트에서는 OAST 기능을 기본 OFF로 둔다. ([GitHub][10])

## 19. 성공 기준

### 19.1 기능 기준

```text
1. scope.yaml 기반으로 in-scope/out-of-scope를 정확히 분리한다.
2. Fast/Normal/Deep profile이 정상 동작한다.
3. raw output을 모두 저장한다.
4. 모든 tool output을 공통 schema로 normalize한다.
5. 중복 URL/finding을 제거한다.
6. BOLA/IDOR 후보를 생성한다.
7. JS endpoint와 secret 후보를 생성한다.
8. API docs와 GraphQL 후보를 생성한다.
9. 위험도 랭킹으로 P0~P3를 생성한다.
10. report.md를 생성한다.
```

### 19.2 안전 기준

```text
1. out-of-scope 대상에는 요청하지 않는다.
2. blocked test는 실행하지 않는다.
3. active exploit은 수행하지 않는다.
4. automatic submission은 수행하지 않는다.
5. secret은 자동 사용하지 않는다.
6. subdomain takeover 자동 claim은 수행하지 않는다.
7. OAST는 기본 OFF이며 명시적 활성화가 필요하다.
```

### 19.3 품질 기준

```text
1. 모든 finding은 source tool, target, confidence, evidence path를 가진다.
2. report.md는 사람이 검증 가능한 manual verification checklist를 포함한다.
3. ranked.jsonl은 priority, risk_score, reason을 포함한다.
4. 동일 finding이 중복으로 여러 번 출력되지 않는다.
5. 로그와 raw output을 통해 재현 가능한 분석 흐름을 제공한다.
```

```text
가장 중요한 개발 포인트:
이 도구의 성능은 nuclei를 많이 돌리는 데서 나오지 않는다.
진짜 핵심은 JS/API/parameter/BOLA·IDOR 후보 생성/랭킹 엔진이다.
따라서 scanner wrapper보다 normalize, candidate engine, risk ranking, evidence/report 품질을 우선 구현해라.

이 요구사항을 기준으로:

순서로 상세 구현 계획과 코드를 작성해라.
```

## 20.1 LLM Adjudication Layer

본 프로젝트는 SCOUT의 LLM tribunal 구조를 웹 블랙박스 버그바운티 자동화에 맞게 적용한다.

외부 도구와 rule-based engine은 deterministic evidence를 생성한다.
LLM은 이 evidence를 기반으로 false positive 감소, impact 판단, 수동 검증 우선순위화, report 초안 생성을 수행한다.

LLM은 취약점을 확정하지 않으며, 자동 exploit이나 자동 제보를 수행하지 않는다.
최종 산출물은 manual verification queue이다.

### LLM Roles

1. Evidence Summarizer
   - raw request/response, scanner output, endpoint metadata를 요약한다.

2. Advocate
   - finding이 보안적으로 의미 있을 가능성을 주장한다.

3. Critic
   - false positive, impact 부족, scope 문제, evidence 부족을 지적한다.

4. Judge
   - Advocate/Critic 결과를 종합해 verdict, confidence, priority, manual verification steps를 JSON으로 출력한다.

### LLM Backends

- Codex CLI primary
- OpenAI API optional fallback
- Claude API optional fallback
- ClaudeCodeCLI optional fallback
- Ollama/local model optional research backend
- No rule-only fallback is allowed

기본 backend는 OpenAI API가 아니라 Codex CLI로 한다.
단, Codex CLI를 파이프라인에서 직접 호출하지 않고 `CodexJudgeDriver`로 감싼다.
backend abstraction은 유지해서 OpenAI API, Claude, local model로 교체 가능하게 한다.
개발/테스트/CI는 외부 호출 실패 시 fail-closed 동작을 검증한다.
LLM verdict가 없는 candidate/finding은 `llm_pending`, `llm_failed`, 또는
`llm_schema_invalid` 상태로 저장하고 final ranking/report에 포함하지 않는다.

#### Codex CLI Mandatory Driver

```text
EvidencePack
   ↓
redaction
   ↓
Codex job workspace
   ↓
codex exec --json --sandbox read-only --output-schema
   ↓
local JSON Schema validation
   ↓
retry / cache / SQLite job log / audit JSONL
   ↓
standard FindingVerdict
```

Codex CLI 사용 시 필수 통제 항목은 다음과 같다.

1. preflight
   - codex binary existence
   - `codex --version`
   - `codex doctor`
   - broken `CODEX_BIN` 차단

2. isolated evidence workspace
   - 각 job마다 별도 workspace 생성
   - redacted evidence만 저장
   - raw secret 저장 금지

3. structured output enforcement
   - `--output-schema` 사용
   - local `jsonschema` 재검증
   - invalid JSON/schema는 retry 후 `llm_schema_invalid`

4. audit and queue metadata
   - stdout JSONL 저장
   - stderr 저장
   - final verdict JSON 저장
   - prompt/input/schema hash 저장
   - SQLite `llm_jobs`, `llm_attempts` 기록

5. fail-closed
   - Codex 실패 시 rule-only verdict 금지
   - Codex schema 실패 시 final report 제외
   - `--allow-external-llm` 없이 실제 Codex 호출 금지

### Mandatory External LLM Layer

본 프로젝트의 핵심 자동화는 외부 LLM 판단 레이어다.

외부 도구들은 evidence를 생성하고, rule-based logic은 scope/safety filtering과 normalization만 수행한다.
취약점 후보 생성, scanner finding triage, false positive 판단, risk ranking, manual verification step 생성, report 작성은 반드시 외부 LLM 호출을 통해 수행한다.

LLM 호출은 optional이 아니다.
LLM verdict가 없는 candidate/finding은 최종 report에 포함하지 않고 `llm_pending` 또는 `llm_failed` 상태로 저장한다.

#### Mandatory LLM Stages

1. Policy Parser
2. Asset Prioritizer
3. Endpoint Interpreter
4. Candidate Generator
5. Scanner Finding Triage
6. Advocate
7. Critic
8. Judge
9. Risk Ranker
10. Report Writer

#### Implemented LLM Stages

현재 코드에 실제 Codex CLI 호출로 구현된 stage는 다음 3개다.

1. Candidate Generator
   - CLI: `donzo candidates generate`
   - 입력: endpoint/API/JS/parameter JSON 또는 JSONL
   - 호출 단위: 개별 URL이 아니라 batch/cluster
   - 출력: manual-review candidate list

2. Scanner Finding Triage
   - CLI: `donzo tribunal run`
   - 입력: finding JSON object 1개
   - 호출 단위: finding 1개
   - 출력: `FindingVerdict`

3. Report Writer
   - CLI: `donzo report draft`
   - 입력: triaged/reviewable finding JSON 또는 JSONL
   - 호출 단위: report draft 1개
   - 출력: Markdown draft

각 stage는 `--allow-external-llm`이 없으면 실제 Codex 호출을 수행하지 않는다.
cache hit, out-of-scope, report 제외 상태는 LLM 호출 0회로 처리한다.
schema 실패나 실행 실패는 `max_attempts`까지 retry한 뒤 fail-closed로 저장한다.

#### Failure Policy

- LLM driver error: mark `llm_failed`
- LLM timeout: retry with backoff
- LLM rate limit: queue batch job
- Invalid JSON/schema: retry, then mark `llm_schema_invalid`
- Rule-based fallback: prohibited
- Final ranking without LLM verdict: prohibited

#### Security / Privacy Policy

- Raw secrets must not be sent to external LLM.
- Cookies, Authorization headers, tokens, API keys are redacted.
- Secret type, source path, hash, and surrounding non-sensitive context may be sent.
- LLM must not be allowed to trigger exploit execution.
- LLM must not automatically validate secrets.
- LLM must not automatically claim subdomain takeover.
- LLM must not submit reports automatically.

### LLM Safety

- no automatic exploit
- no automatic submission
- no secret validation
- no subdomain takeover claim
- no destructive testing
- no credential attack
- manual verification required

### Verdict

```text
confirmed_candidate
likely_true_positive
needs_manual_review
likely_false_positive
out_of_scope_or_not_allowed
```

어떤 verdict도 자동 제보나 취약점 확정을 의미하지 않는다.
